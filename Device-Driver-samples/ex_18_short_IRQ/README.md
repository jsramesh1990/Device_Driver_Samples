# **SHORT Interrupt Handler Driver**

## **PROJECT OVERVIEW**
This Linux kernel module demonstrates hardware interrupt handling. It connects to an I/O port (default: parallel port at 0x200) and handles interrupts (IRQ 6) generated by hardware events. When hardware triggers an interrupt, the driver counts it and makes the count available to user programs through `/dev/short`. This is essential learning for real-time hardware interaction.

## **QUICK START**

### **Build & Install:**
```bash
# Build the module
make

# Load with default IRQ 6 and port 0x200
sudo insmod short_IRQ.ko

# Load with custom parameters
sudo insmod short_IRQ.ko short_irq=7 base=0x378
```

### **Basic Usage:**
```bash
# Read interrupt count
cat /dev/short

# Write to hardware (may trigger interrupts)
echo -n "data" | sudo dd of=/dev/short

# Monitor interrupt activity
sudo dmesg | tail -20
```

## **HOW INTERRUPTS WORK**

### **Simple Explanation:**
Interrupts are like "doorbells" from hardware:
```
Hardware Event → "Ding!" (Interrupt) → Driver Wakes Up → Processes Event
```

Without interrupts, the CPU would have to constantly check hardware ("polling"), wasting resources. With interrupts, hardware tells the CPU when it needs attention.

## **DETAILED WORKING FLOW**

### **1. MODULE INITIALIZATION**
```
Command: sudo insmod short_IRQ.ko
    |
    v
m_init() function:
1. Allocate device structure with spinlock
2. Request I/O port region (0x200)
3. Register interrupt handler for IRQ 6:
   - request_irq(6, irq_service, 0, "short", dev)
4. Enable hardware interrupts:
   - outb(1, 0x201) // Write to control port
5. Register character device (/dev/short)
6. Create /dev/short device file
    |
    v
Driver ready to handle interrupts
```

### **2. INTERRUPT HANDLING**
```
Hardware Event (e.g., button press):
    |
    v
1. Hardware activates IRQ line 6
2. CPU receives interrupt, saves context
3. Kernel calls registered handler: irq_service()
    |
    v
irq_service() executes:
1. Verify correct IRQ: if (short_irq != irq)
2. Lock spinlock: spin_lock_irqsave()
3. Increment interrupt counter: dev->cnt++
4. Log debug message: "IRQ triggered X times"
5. Unlock spinlock: spin_unlock_irqrestore()
6. Return IRQ_HANDLED to kernel
    |
    v
Kernel restores saved context
CPU resumes previous task
```

### **3. USER READING INTERRUPT COUNT**
```
Command: cat /dev/short
    |
    v
1. read() system call
2. Driver's short_read() called
3. Lock spinlock for safe access
4. Read dev->cnt value
5. Unlock spinlock
6. Copy count to user buffer
7. Return count to user
    |
    v
User sees: "123" (123 interrupts handled)
```

### **4. USER WRITING TO HARDWARE**
```
Command: echo "data" > /dev/short
    |
    v
1. write() system call
2. Driver's short_write() called
3. Copy data from user to kernel buffer
4. Write each byte to I/O port (0x200)
5. Memory barrier: wmb()
6. Free kernel buffer
7. Return bytes written
    |
    v
Data sent to hardware
(Might trigger interrupts if hardware configured)
```

### **5. MODULE CLEANUP**
```
Command: sudo rmmod short_IRQ
    |
    v
m_exit() function:
1. Disable hardware interrupts: outb(0, 0x201)
2. Unregister character device
3. Free interrupt: free_irq(6, dev)
4. Release I/O port region
5. Free device structure
6. Remove /dev/short
    |
    v
All resources cleaned up
```

## **TECHNICAL IMPLEMENTATION DETAILS**

### **Key Data Structures:**
```c
struct short_dev {
    spinlock_t lock;     // Protects concurrent access
    int cnt;            // Interrupt counter
};
```

### **Critical Functions:**

#### **Interrupt Handler Registration:**
```c
result = request_irq(short_irq,      // IRQ number
                     irq_service,    // Handler function
                     0,              // Flags
                     MODULE_NAME,    // Device name
                     dev);           // Device data
```

#### **Interrupt Handler:**
```c
irqreturn_t irq_service(int irq, void *dev_id)
{
    struct short_dev *dev = (struct short_dev*)dev_id;
    unsigned long flags;
    
    spin_lock_irqsave(&dev->lock, flags);
    dev->cnt++;  // Critical section: increment counter
    spin_unlock_irqrestore(&dev->lock, flags);
    
    return IRQ_HANDLED;
}
```

#### **Safe Counter Access:**
```c
// In short_read():
spin_lock_irqsave(&dev->lock, flags);
val = dev->cnt;          // Safe read
spin_unlock_irqrestore(&dev->lock, flags);
```

### **Hardware Control:**
```c
// Enable interrupts: write 1 to control port (base + 1)
outb(1, short_base + 1);

// Disable interrupts: write 0 to control port
outb(0, short_base + 1);
```

# test_interrupts.py

import time
import struct

print("Interrupt Driver Test")
print("=" * 40)

try:
    # Open device
    with open('/dev/short', 'rb+') as dev:
        # Read initial count
        initial = struct.unpack('i', dev.read(4))[0]
        print(f"Initial interrupt count: {initial}")
        
        # Monitor for 10 seconds
        print("Monitoring for interrupts...")
        for i in range(10):
            time.sleep(1)
            dev.seek(0)
            current = struct.unpack('i', dev.read(4))[0]
            if current > initial:
                print(f"Interrupt detected! Total: {current}")
                initial = current
        
        print("Test complete")

    print("ERROR: Need root privileges (use sudo)")
except FileNotFoundError:
    print("ERROR: /dev/short not found. Load module first.")

        }

    }
    
    close(fd);
    return 0;
}

## **SAFETY AND BEST PRACTICES**

### **DO in Interrupt Handlers:**
- Keep handlers short and fast
- Use spinlocks for synchronization
- Return proper status (IRQ_HANDLED/IRQ_NONE)
- Acknowledge interrupts to hardware
- Use shared interrupts appropriately

### **DON'T in Interrupt Handlers:**
- Sleep or block
- Access userspace memory
- Perform lengthy operations
- Forget to release locks
- Ignore interrupt status

### **Concurrency Protection:**
```c
// ALWAYS use locking when accessing shared data
spin_lock_irqsave(&dev->lock, flags);
// Critical section
spin_unlock_irqrestore(&dev->lock, flags);

// Disables local interrupts during lock
// Prevents deadlocks with interrupt handler
```

## **ADVANCED TOPICS**

### **1. Interrupt Sharing:**

// Request IRQ with sharing flags

### **2. Bottom Halves (Deferred Work):**
```c
// For lengthy interrupt processing:
// 1. Quick acknowledgment in top half (interrupt handler)
// 2. Schedule workqueue/tasklet for actual processing

struct work_struct my_work;
INIT_WORK(&my_work, process_data);

// In interrupt handler:
schedule_work(&my_work);
```

### **3. Interrupt Request Options:**
```c

IRQF_SHARED      // Share interrupt line
IRQF_TRIGGER_RISING  // Interrupt on rising edge
IRQF_TRIGGER_FALLING // Interrupt on falling edge
IRQF_TRIGGER_HIGH    // Interrupt on high level
IRQF_TRIGGER_LOW     // Interrupt on low level
IRQF_ONESHOT     // Don't re-enable after handling
```

## **PERFORMANCE CONSIDERATIONS**

### **Interrupt Latency:**
- Time from hardware event to handler execution
- Affected by: interrupt masking, handler length, system load
- Critical for real-time applications

### **Optimization Tips:**
1. Keep handlers minimal (just acknowledge + schedule work)
2. Use appropriate interrupt triggering (edge vs level)
3. Consider interrupt coalescing for high-rate events
4. Profile with `perf` or `ftrace`
